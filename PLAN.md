# 程序员转 AI 应用开发——路线拆解工具

## 项目定位
求职作品集，展示三项核心能力：**RAG + AI 编排 + 测试**。

- 技术栈：Flask + React，LLM 用 DeepSeek API（openai 包 + base_url 指向 deepseek）
- 第一优先级：求职。不让产品功能挤掉求职进度。

---

## 核心逻辑：两步链式编排

| 步骤 | 名称 | 输入 | 输出 |
|------|------|------|------|
| 1 | 收敛 | 现有技能 / 目标程度 / 时长 | 一个可衡量目标 |
| 2 | 生成 | 收敛目标 + RAG 检索到的转型知识 | 分阶段路线（每阶段带"可执行的下一步指引"） |

## 五周计划

### 第1周：跑通链式编排（无 RAG、无前端）✅ 已完成
- ① API 调通 ✅
- ② 写「收敛目标」prompt（三输入 → 一个可衡量目标）✅
- ③ 写「生成路线」prompt + 强制 JSON 输出 ✅
- ④ 两步串成链 ✅
- ⑤ 套 Flask 接口 ✅

### 第2周：接入 RAG

动作①——搞懂 RAG 原理 + 跑通伪 RAG（✅ 今天已完成）

理解检索增强生成是什么、为什么能治幻觉；手搓"整段全塞"的最小版，验证了"有据才答、无据不编"。

动作②——搞懂向量检索，跑通最小向量化 ✅ 已完成

理解"为什么需要检索"（整段全塞的局限你今天已经看到了）、向量怎么表示语义。然后用一个 embedding 接口，把几段文字转成向量、能算出"谁和谁更相近"。先不碰向量库，只验证"文字→向量→比相似度"这条路通。

动作③——把检索接进来，做成真 RAG ✅ 已完成

用户提问 → 只检索出最相关的几段 → 塞进 prompt 回答（而不是全塞）。这一步把动作①的"全塞"换成"按需检索"，项目就从"伪 RAG"变成"真 RAG"了。最小实现可以先不上专业向量库，用最朴素的相似度计算；通了再决定要不要换成向量库工具。

动作④——把 RAG 接进你现有的链/接口 ✅ 已完成

让 RAG 能力和第1周做的 Flask 接口、两步链结合——比如生成路线时能参考知识库。这一步是"缝合"，对应上周动作④把零件串起来的角色。

动作⑤——记录 eval 发现 + 收尾 ✅ 已完成

RAG 跑起来后，observe 它哪里不可靠（检索捞错段？答非所问？），记进 PLAN.md——这又是第3周 eval 的真实弹药，也是面试讲"我发现了 RAG 哪里不靠谱"的料。

**动作⑤ eval 发现记录：**

1. **编码坑**：PowerShell 发中文请求变问号，需用 UTF-8 字节 + `charset=utf-8` 才能正确传递中文。这是 Windows 环境下 Flask API 调用的实际踩坑。

2. **eval 发现①：假 embedding 缺陷在小知识库下被掩盖**。当前知识库只有 7 条，检索压力太小，即使用字符重合度（假 embedding）也能碰巧命中。需要扩大知识库才能真正测出检索质量差异。

3. **eval 发现②：prompt"无据不编"约束有效**。喂了干扰段落给模型，模型没有硬编，确实遵守了"资料中没有就说不清楚"的指令。这验证了 RAG 对抗幻觉的基本机制是可靠的。

4. **eval 发现③：假 embedding 不懂语义、只看字符重合**。当用户问题和知识段落用词不同但语义相同时，假 embedding（基于字符匹配）检索失败、会捞回不相关段落。这是"为什么必须换真 embedding"的直接证据，面试时可以用具体例子讲清楚。

### 第3周：测试体系——差异化王牌

动作①——单元测试入门 + 给一个纯函数写第一个测试

你有软件测试背景，但那是手工/功能测试。这里是代码层面的自动化单元测试（用 pytest）。先挑一个最"干净"、不依赖 AI 的函数练手——比如 step7 里的 retrieve 或 fake_score，它们输入输出确定，最适合写第一个测试，建立"写测试→自动跑→看红绿"的肌肉记忆。

动作②——搞懂"LLM输出怎么测"这个难点

传统测试是"输入A就该输出B"，但 LLM 每次输出都不一样，没法用 == 判对错。这一步是理解 eval 和普通测试的根本区别、有哪些评估思路（规则校验、用另一个LLM打分等）。先理解，不写代码。

动作③——把你的真实发现写成自动化校验

挑 PLAN.md 里最好测的那条（比如"各阶段时长之和不能超总时长"——这是个数值规则，能用代码硬判），写成一个能自动跑的检查：喂模型输出进去，自动判断它有没有违反约束。这是 eval 体系的第一块真砖。

动作④——批量跑 + 出报告

不止测一次。准备几条测试输入，让它们批量跑过你的链路/RAG，自动统计"几条通过、几条违反约束"，输出一个简单结果。这就有了"评估报告"的雏形。

动作⑤——整理成求职可讲的材料 + 收尾

把这套 eval 整理成能在面试讲、能写进 README 的东西：我发现了哪些问题→我怎么设计测试去验证→结果如何。commit、更新 PLAN.md。

## 已知问题 / 待调优

| # | 问题 | 计划处理节点 |
|---|------|-------------|
| 1 | 各阶段时长之和会超过用户输入的总时长 | 第3周 eval 中作为重点测试 case |
| 2 | ~~LLM 输出是裸数组 `[...]`，不是带 `{phases: [...]}` 的阶段列表对象~~ ✅ 已解决 | 动作④通过强制 JSON + 后处理统一为 `{phases: [...]}` 结构 |
| 3 | 数值漂移 + 重点错配叠加：目标时间约束=1个月，但生成路线各阶段时长之和≈20周（4+4+6+6周+2天），超约束约5倍；且核心诉求是"RAG应用"，模型却把5个阶段里4个堆在传统前后端（Python/HTML/Vue/Flask），真正AI部分被压成第5阶段"2天突击" | 第3周 eval 高价值 case（单条同时覆盖两个失败模式：数值约束不可靠 + 多约束下权重错配） |
| 4 | 指标与动机冲突时，模型优先指标：当指标很赶（如3个月出上线作品）、动机却是"体系化转行"时，模型为满足指标牺牲了动机，路线偏实战而非偏基础；把"指标松、时间足"后，动机分支正常生效，说明多约束间存在优先级 | 第3周 eval 可研究这个优先级行为 |
| 5 | 字段名跨次运行不稳定：同一路线生成 prompt，多次运行字段名在中文（如`下一步指引`）和英文驼峰（如`nextStep`）之间漂移，前端/下游按固定字段名取值时会取空报错 | 第3周 eval 加字段名校验；prompt 中钉死字段名，解析后加一层标准化 |
| 6 | 编码坑：PowerShell 发中文请求变问号，Windows 默认编码非 UTF-8，Flask 收到的 request body 中文损坏 | 已解决：发送端用 UTF-8 字节 + `charset=utf-8`；记入踩坑文档供面试"工程落地踩坑"叙事 |
| 7 | eval 发现①：假 embedding（字符匹配）在小知识库下缺陷被掩盖，检索压力太小，碰巧命中率高 | 第3周：扩大知识库后再测检索质量，对比真/假 embedding 的命中率差异 |
| 8 | eval 发现②：prompt"无据不编"约束有效，喂干扰段落模型未硬编——RAG 对抗幻觉的基本机制可靠 | 第3周可设计边界 case 专门压测这个约束的极限（如喂矛盾段落、喂半相关段落） |
| 9 | eval 发现③：假 embedding 不懂语义、只看字符重合，同义不同词时检索失败，捞回不相关段落 ；面试时可用具体例子对比"换之前 vs 换之后"展示理解深度 |
| 10 | `generate_route` 对非 JSON 返回无兜底，LLM 偶尔输出非 JSON 文本时会抛 `JSONDecodeError` 崩溃，整个请求失败 | 已用 `test_generate_route_dirty_return` 锁定现状；待加容错——解析失败时返回安全默认值或自动重试 |
| 11 | `generate_route` 未校验阶段数，LLM 返回 2 个（或 >6 个）阶段也会原样收下，违反 prompt 中"3-6 个阶段"的约束 | **动作3-B**：`json.loads` 后加 3-6 范围校验，不满足抛异常或触发重试；`test_generate_route_too_few_stages` 已锁定现状（断言 2 个阶段通过），校验加入后需同步改为 `pytest.raises` |

## 设计决策：每阶段带"可执行的下一步指引"

生成路线时，每个阶段附带一句具体可执行的下一步动作。几乎零成本实现，且强化面试时的产品 sense 叙事。
