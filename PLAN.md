# 程序员转 AI 应用开发——路线拆解工具

## 项目定位
求职作品集，展示三项核心能力：**RAG + AI 编排 + 测试**。

- 技术栈：Flask + React，LLM 用 DeepSeek API（openai 包 + base_url 指向 deepseek）
- 第一优先级：求职。不让产品功能挤掉求职进度。

---

## 核心逻辑：两步链式编排

| 步骤 | 名称 | 输入 | 输出 |
|------|------|------|------|
| 1 | 收敛 | 现有技能 / 目标程度 / 时长 | 一个可衡量目标 |
| 2 | 生成 | 收敛目标 + RAG 检索到的转型知识 | 分阶段路线（每阶段带"可执行的下一步指引"） |

---

## 五周计划

### 第1周：跑通链式编排（无 RAG、无前端）
- ① API 调通 ✅
- ② 写「收敛目标」prompt（三输入 → 一个可衡量目标）
- ③ 写「生成路线」prompt + 强制 JSON 输出
- ④ 两步串成链
- ⑤ 套 Flask 接口

### 第2周：接入 RAG
- 准备转型知识文档
- 向量化 + 检索（embedding + ChromaDB / FAISS）
- 将检索结果注入「生成路线」prompt

### 第3周：测试体系——差异化王牌
- 单元测试（Flask 路由 + 编排逻辑）
- LLM 输出 eval（结构校验 + 内容质量评分）

### 第4周：React 前端串联
- 用户输入表单（三输入）
- 路线结果展示（时间线 / 阶段卡片）
- 与 Flask API 对接

### 第5周：打磨 + README + 求职化
- README 突出三项能力 + 设计决策
- 录屏演示
- 代码清理与注释

## 已知问题 / 待调优

| # | 问题 | 计划处理节点 |
|---|------|-------------|
| 1 | 各阶段时长之和会超过用户输入的总时长 | 第3周 eval 中作为重点测试 case |
| 2 | LLM 输出是裸数组 `[...]`，不是带 `{phases: [...]}` 的阶段列表对象 | 动作④「两步串成链」时统一输出结构 |
 **指标与动机冲突时，模型优先指标** —— 当指标很赶（如3个月出上线作品）、动机却是"体系化转行"时，模型为满足指标牺牲了动机，路线偏实战而非偏基础。把"指标松、时间足"后，动机分支正常生效。说明多约束间存在优先级。eval 可研究这个优先级行为。


---

## 设计决策：每阶段带"可执行的下一步指引"

生成路线时，每个阶段附带一句具体可执行的下一步动作。几乎零成本实现，且强化面试时的产品 sense 叙事。
